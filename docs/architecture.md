# SkyMind Technical Architecture

This document outlines the technical design decisions and data flows for the SkyMind Tier 1 Proof of Concept.

## Design Philosophy
1. **Serverless First**: Minimize fixed costs. By using Lambda, API Gateway, and DynamoDB (On-Demand), the baseline cost of running SkyMind is effectively zero. You only pay for active processing.
2. **Decoupled Agents**: The AI analysis is decoupled from the execution (Healer). This provides a security boundary where the Analyzer can safely analyze data without risking unintended mutations to the infrastructure.
3. **Event-Driven**: Agents run on EventBridge chron schedules, reducing continuous compute overhead.

## Data Model (DynamoDB)

We use DynamoDB for high read/write throughput and infinite scaling at a low cost.

### `skymind-resources` Table
Stores the inventory of all discovered AWS resources.
- `id` (PK): E.g., `i-0abcdef12345` or `my-lambda-func`
- `type`: `EC2`, `Lambda`, `RDS`, `S3`, etc.
- `name`: Tag-based name or resource identifier.
- `status`: `healthy`, `warning`, `critical`
- `region`: E.g., `us-east-1`
- `lastUpdated`: ISO-8601 timestamp

### `skymind-metrics` Table
Stores time-series metrics.
- `id` (PK): Resource ID.
- `timestamp` (SK): ISO-8601 timestamp (allows querying historical data for a resource).
- `cpu`, `memory`, `latency`, `errorRate`, etc.
- `anomaly`: Boolean.
- `anomalyDetails`: String describing the anomaly if detected.

### `skymind-alerts` Table
Stores active and resolved incidents generated by the AI Analyzer.
- `id` (PK): Alert UUID.
- `resourceId`: The affected resource.
- `timestamp`: Creation time.
- `severity`: `info`, `warning`, `critical`.
- `title`, `description`: Context generated by Bedrock.
- `recommendedAction`: Step required to resolve.
- `autoHealAllowed`: Boolean flag authorizing the Healer Lambda to act.
- `resolved`: Boolean.

## Lambda Agent Workflows

### 1. Scanner Agent (`scanner/index.js`)
- **Triggered By**: EventBridge (Every 10 min)
- **Permissions**: `ec2:DescribeInstances`, `lambda:ListFunctions` 
- **Action**: Sweeps the account, catalogs resources, and upserts them into `skymind-resources`.

### 2. Metrics Agent (`metrics/index.js`)
- **Triggered By**: EventBridge (Every 5 min)
- **Permissions**: `cloudwatch:GetMetricStatistics`
- **Action**: Reads the resource table, queries CloudWatch for the last 5 minutes of data for each resource. Performs basic statistical anomaly detection (e.g., Z-score, static thresholds). Upserts results to `skymind-metrics`.

### 3. AI Analyzer (`analyzer/index.js`)
- **Triggered By**: EventBridge (Every 5 min)
- **Permissions**: `bedrock:InvokeModel`
- **Action**: Queries `skymind-metrics` for recent anomalies. Formats a prompt containing the anomalies and asks Claude 3 Haiku to diagnose the severity and recommend an action. Inserts resulting JSON into `skymind-alerts`.

### 4. Auto-Healer (`healer/index.js`)
- **Triggered By**: EventBridge (Every 5 min)
- **Permissions**: Varies (Requires write permissions to managed resources, e.g., `ec2:RebootInstances`).
- **Action**: Queries unresolved alerts where `autoHealAllowed = true`. Executes the recommended action. Updates the alert to `resolved = true` and logs the outcome.

### 5. NL Ops Chat API (`chat/index.js`)
- **Triggered By**: API Gateway (User input)
- **Permissions**: `bedrock:InvokeModel`
- **Action**: Uses naive RAG (Retrieval-Augmented Generation). Scans the `skymind-resources` and `skymind-alerts` tables to build an infrastructure context block. Appends the user's question, sends to Claude, and streams the response back to the client.

## Security Controls

If SkyMind were to be deployed in a production enterprise environment, the following controls are necessary:
1. **Cross-Account Roles**: SkyMind should ideally run in a dedicated security/tooling AWS account and assume roles into target workload accounts to read metrics.
2. **Healer Boundaries**: The Healer IAM role must have an explicit `Deny` policy on actions like `iam:*`, `kms:*`, and `cloudtrail:*`.
3. **Approval Workflows**: Integration with AWS Step Functions to route high-risk actions through Slack or Email for human approval before execution. (Simulation in POC).
